**Admin page** (current state in `frontend/src/routes/admin.tsx`):
- Shows: system status card (health badge, DB status, version), workers table (ID, status, current task), log viewer.
- Config section is stubbed as "Coming soon".
- Backend has `GET /admin/config` but it returns **mock data** — not wired to real config.
- `GET /status` returns rich system info: memory, goroutines, DB connection details, scan queue stats.
- `GET /health` returns dependency checks (DB, metrics, scan queue, memory, goroutines).
- No settings persistence — everything is read-only from `config.yaml`.

## Data Model Decisions

### Tags: Array column vs. join table

**Recommendation: `TEXT[]` array column on hosts table.**

Rationale: tags are simple string labels, the host count is bounded (thousands, not millions), PostgreSQL `TEXT[]` supports `@>` (contains) and GIN indexing for fast lookups, and this avoids the complexity of a separate `tags` + `host_tags` join table. The existing `HostRequest.Tags` already models them as `[]string`.

### Groups: Separate table with membership

**Recommendation: `host_groups` table + `host_group_members` join table.**

Rationale: groups have their own metadata (name, description, optional filter rule), a host can belong to multiple groups, and groups need their own CRUD endpoints. A join table keeps group membership queryable without JSONB gymnastics.

```sql
-- Migration 008
ALTER TABLE hosts ADD COLUMN tags TEXT[] DEFAULT '{}';
CREATE INDEX idx_hosts_tags ON hosts USING GIN (tags);

CREATE TABLE host_groups (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(100) NOT NULL UNIQUE,
    description TEXT DEFAULT '',
    filter_rule JSONB,          -- optional: auto-membership via FilterExpr
    color       VARCHAR(7),     -- hex color for UI badge
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE host_group_members (
    host_id     UUID REFERENCES hosts(id) ON DELETE CASCADE,
    group_id    UUID REFERENCES host_groups(id) ON DELETE CASCADE,
    added_at    TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (host_id, group_id)
);
```

## Risks & Open Questions

1. **Tag storage: array vs. join table.** This plan uses `TEXT[]` for simplicity. If tags need to carry metadata later (created_by, color, description), a join table would be better. The API shape (`[]string`) stays the same either way, so this is a backend-only migration if needed.

2. **Auto-membership groups.** The `filter_rule` field on groups is included in the schema but not actively enforced in v0.24. Auto-membership (groups that automatically include hosts matching a filter) is deferred — it adds complexity around when to re-evaluate membership and how to handle manual overrides. The field is there so we don't need a schema migration later.

3. **Tag colors.** This plan doesn't assign colors to tags (tags are plain strings). If colored tags are wanted, that's a separate `tag_definitions` table — better as a follow-up.

4. **Scan tags.** The roadmap mentions tags on scans. This plan focuses on host tags only, which is the higher-value feature. Scan tags can reuse the same `TagInput` component and follow the same pattern in a later PR.

5. **Settings mutability boundary.** Some settings (DB connection, API port, TLS) can't safely change at runtime. The plan marks these as restart-required with a UI badge. The boundary between mutable and immutable settings should be reviewed before implementation — getting it wrong could cause subtle bugs. In particular, `LoadSettingsOverrides` failing at startup would prevent the app from booting — wrap it carefully.

6. **Stats endpoint performance.** `GET /api/v1/stats/summary` runs multiple aggregation queries. If the hosts table grows large, these may need caching or materialized views. Start without caching and measure.

7. **Activity feed message volume.** A large scan that updates progress every second could flood the feed. The frontend deduplicates progress ticks (only showing state transitions), but the backend should also consider throttling WebSocket broadcasts during high-activity periods.
